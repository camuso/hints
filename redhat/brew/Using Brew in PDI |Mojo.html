    Using Brew in PDI

    Version 28
    Created by Michael Bonnet on Mar 15, 2016 1:56 PM. Last modified by Michael McLean on Apr 20, 2016 4:04 PM.

        • Overview
        • Web interface
        • Command line
        • API
        • Message Bus

      Overview

      As of April 5, 2016, Brew has migrated into PnT DevOps Infrastructure. This will allow for simplified management and better ability to scale resources as necessary. This document describes the changes necessary to access Brew in the new infrastructure.

       

      Web interface

      The web interface for Brew is: https://brewweb.engineering.redhat.com/brew/

       

      Command line

      1. Grab the appropriate .repo file for your system and install it in /etc/yum.repos.d/, e.g.
        • RHEL 6 Server - rcm-tools-rhel-6-server.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-6-server.repo
        • RHEL 6 Client - rcm-tools-rhel-6-client.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-6-client.repo
        • RHEL 6 Workstation - rcm-tools-rhel-6-workstation.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-6-workstation.repo
        • RHEL 7 Server - rcm-tools-rhel-7-server.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-7-server.repo
        • RHEL 7 Client - rcm-tools-rhel-7-client.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-7-client.repo
        • RHEL 7 Workstation - rcm-tools-rhel-7-workstation.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/RCMTOOLS/rcm-tools-rhel-7-workstation.repo
        • Fedora 22, 23, 24, and 25 - rcm-tools-fedora.repo
          curl -L -O http://download.devel.redhat.com/rel-eng/internal/rcm-tools-fedora.repo
      2. Install/update the koji and brewkoji packages
        1. yum install koji brewkoji  
        yum install koji brewkoji
        or
        1. yum update koji brewkoji  
        yum update koji brewkoji
        Some CSB users have been unable to update because the CSB uses the yum priorities plugin. To address this, we have added a high priority setting to the rcm-tools-rhel* repo files listed above. If you previously downloaded an earlier version of the repo file, you may want to download the current version.
      3. Ensure you have the latest version of the packages installed, koji >= 1.10.1 and brewkoji >= 1.12
        1. $ rpm -q koji brewkoji  
        2. koji-1.10.1-22.3.el7.noarch  
        3. brewkoji-1.12-1.el7.noarch  
        $ rpm -q koji brewkoji
koji-1.10.1-22.3.el7.noarch
brewkoji-1.12-1.el7.noarch
      4. Verify
        1. brew hello  
        brew hello
      5. If you use the rhpkg command, update the rhpkg package as well
        1. yum update rhpkg  
        yum update rhpkg

      If you have a [brew] entry in ~/.koji/config, it may interfere with rhpkg, and result in a

      Server not found in Kerberos database

      error. Either comment that section out, our update it to match the configuration in the API section below.

      The brew and rhpkg command line tools are now communicating with the new Brew instance.

      If you receive an error message like:

      Could not connect to Kerberos authentication service: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed (_ssl.c:590)

      make sure you have imported the Red Hat IS CA certificate onto your machine. See Specification: Recommended System-level PKI Settings for instructions.

      If the brew command appears to stall with no output in Fedora, it is likely that you are also experiencing a certificate verification issue. Try importing the Red Hat IS CA certificate as described above.

       

      API

      Folks who prefer to manage their Brew configuration themselves should use the following parameters (usually in /etc/koji.conf or ~/.koji/config):

      1. server = https://brewhub.engineering.redhat.com/brewhub  
      2. topurl = http://download.eng.bos.redhat.com/brewroot  
      3. weburl = https://brewweb.engineering.redhat.com/brew  
      4. krbservice = brewhub  
      5. authtype = kerberos  
      server = https://brewhub.engineering.redhat.com/brewhub
topurl = http://download.eng.bos.redhat.com/brewroot
weburl = https://brewweb.engineering.redhat.com/brew
krbservice = brewhub
authtype = kerberos

      If you're using the API directly, and all access is in read-only mode, changing the hub and web URLs will be the only change required.

      If you use Kerberos to log in to your session so that you can launch tasks or otherwise perform data-modifying operations, you'll need to pass krbservice into your session object at creation time.

      #!/usr/bin/python

      import koji

      session = koji.ClientSession('https://brewhub.engineering.redhat.com/brewhub', opts={'krbservice': 'brewhub'})

      session.krb_login()

      If you receive an error message like:

      Decrypt integrity check failed

      make sure you're using the correct krbservice (brewhub rather than host). Your config file should contain:

      krbservice = brewhub

      If you receive an error message like:

      Server not found in Kerberos database

      make sure you have your krbservice set correctly (see above). It may also be caused by the CNAME of the service not being resolved to the canonical hostname when performing Kerberos authentication. Make sure you have the very latest version of the koji package installed (version >= 1.10.1). If you're accessing the XMLRPC API using another method, you'll need to do the forward/reverse DNS mapping to translate the CNAME to the canonical hostname. In python, socket.getfqdn() can handle this.

      Message Bus

      Messages produced by the Brew instance are sent to qpid.devel.redhat.com. Have your AMQP client connect to:

      amqps://qpid.devel.redhat.com

      Messages are sent to the brew.events exchange, and Kerberos authentication is supported.

          • Alexander Bokovoy Apr 5, 2016 3:22 AM

            If you are remote and use DNS split configuration to re-route DNS requests to Red Hat DNS servers with dnsmasq, you need to make sure reverse zone requests also come to Red Hat's DNS servers.

             

            This can be achieved with '--rev-server' option of dnsmasq:

             

            --rev-server=10.0.0.0/8,10.38.5.26

             

            will make sure 10.0.0.0/8 will get resolved via Red Hat's DNS server.

             

            The reason for this is because Python's socket.getfqdn() call actually uses socket.gethostbyaddr() which will happily report 'unknown host' for any host that has no PTR record resolved.
